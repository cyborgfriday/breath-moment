# GitHub 專案發布注意事項與新手教學

恭喜您將專案發布到 GitHub！這是一個很大的進步。既然您的專案是公開的（Public），這意味著全世界的人都可以看到您的程式碼。以下是您作為新手應該注意的所有重要事項：

## 1. 如果我不想被別人拿走，該怎麼辦？
如果您覺得自己的代碼非常有價值，不希望被任何人看見或利用，有兩個最直接的解決方法：

### 方法 A：將專案轉為「私有 (Private)」
這是最徹底的保護方式。
*   **私有專案**：只有你自己（以及你邀請的人）看得到代碼。
*   **注意 GitHub Pages**：在免費帳號下，**如果您將專案設為 Private，GitHub Pages (您的公開網頁) 就會無法公開發布**（或者需要付費升級）。
*   **解決方案**：如果您想「代碼私有」但「網頁公開」，建議改用 **Vercel** 或 **Netlify** 進行部署。它們可以連結您的私有 GitHub 專案，但生成的網頁網址仍然是全世界公開的。

### 方法 B：選擇嚴格的授權 (License)
如果你仍想公開但希望限制別人的用途：
*   **不加 License**：這在法律上最強硬，因為預設「保留所有權利」，沒人能合法使用你的代碼。但這也是雙面刃，會讓正面合作的人也無法靠近。
*   **嚴格 License (如 GPL-3.0)**：規定只要用了你的代碼，對方的專案也必須開源，這常讓想用於商業競爭的人卻步。

## 2. 隱私與安全：誰看得到什麼？
*   **全部原始碼**：只要在 GitHub 上，任何人都可以閱讀、下載您的程式碼。
*   **提交紀錄 (Commits)**：歷史紀錄中曾經上傳過的東西，即使後來刪掉了，在 Git 歷史中還是找得到。
*   **敏感資訊 (Secrets)**：**千萬不要**把密碼、API 金鑰、個人通訊錄等上傳。您目前的 `.gitignore` 有排除 `.env` 文件，這是正確的做法。

## 3. AI 協作的痕跡會被看到嗎？
如果您使用 AI 助手（如 Claude 或其他編碼助手）開發，GitHub 上會看到以下內容：
*   **提交者名稱**：有些工具會自動以 AI 的名義提交（例如您的專案中出現了 `claude` 作為貢獻者），這會讓別人知道您使用了 AI。
*   **提交訊息 (Commit Messages)**：如果您讓 AI 自動寫 Commit，訊息中可能會包含「由 AI 生成」或類似的描述。
*   **特有的資料夾**：包含 `.agent`、`.agents`、`.claude`、`.kiro` 等隱藏目錄。這些目錄存放的是 AI 的「技能」或設定。**目前我們已將它們加入 `.gitignore`，這樣它們就不會被上傳到 GitHub 了。**
*   **修改過程**：Git 會記錄「每一行代碼的變化」。雖然看不到 AI 是如何「想」的，但可以看到代碼從 A 變成 B 的演進過程。

## 4. 公開代表「每個人都可以修改我的檔案」嗎？
**不完全是。** 這是新手最容易擔心的問題：
*   **他們不能直接改您的檔案**：除非您把對方加為「Collaborator」（協力者），否則任何人都無法直接把程式碼推送到您的 Repository 中。
*   **Fork (分支副本)**：別人可以點擊 "Fork" 按鈕，把您的程式碼複製一份到**他們自已的帳號下**隨意更改，但這不會影響到您原本的程式碼。
*   **Pull Request (合併請求)**：如果別人在他們的副本中改得很好，想幫您改 Bug，他們可以發起 "Pull Request"。這時候**只有在您點選同意（Merge）後**，他們的修改才會進到您的專案裡。

## 5. 為什麼大家要公開程式碼？
這是一個很棒的哲學問題。公開程式碼（Open Source）有幾個核心目的：
*   **技術交流與學習**：很多開發者都是透過看別人的程式碼學習的。
*   **建立個人品牌（作品集）**：對雇主或合作夥伴來說，GitHub 是最好的「技術名片」。
*   **群策群力**：當專案變大時，全世界的牛人都可以幫您修 Bug 或增加新功能。
*   **社群回饋**：很多開發者使用了別人的免費工具，也會想把自己寫的有價值的東西回饋給社群。

## 6. 專案的門面：README.md
README 是別人點進您專案第一個看到的內容。
*   **它包含了什麼？**：專案是什麼、怎麼安裝、怎麼執行。
*   **建議補充**：增加開發指令（例如 `npm install` 與 `npm run dev`），讓想嘗試或幫忙的人知道從何開始。

## 7. 法律與授權：LICENSE
您的 `breath-moment` 目前使用的是 **CC BY-NC-SA 4.0 (創用 CC 姓名標示-非商業性-相同方式分享)**。

這是一個非常嚴格的授權，符合您的兩大要求：
*   **禁止商用 (Non-Commercial)**：任何人不得將您的代碼用於商業營利目的。
*   **強制開源 (ShareAlike)**：如果有人修改了您的代碼，他們的新版本也必須以同樣的「非商業、開源」方式分享出來。
*   **姓名標示 (Attribution)**：別人使用時必須註明原作者是您。

> [!NOTE]
> 這種授權雖然能極大程度保護您的權利，但也會讓一些想要合法「贊助」或「整合」您的企業遠離。如果您未來想將其轉為商業產品，隨時可以更改授權（因為版權始終在您手裡）。

## 8. 專業感的小細節
*   **`package.json` 的資訊**：我們已經將名稱從 `"code-to-figma"` 修正為 `"breath-moment"`。
*   **提交訊息 (Commit Messages)**：盡量寫得有意義，例如 `fix: 修正時間顯示錯誤` 而不是 `update 1`。

## 10. 身為設計師：GitHub 是您的「超能力擴展」
對於一位設計師來說，能直接在 GitHub 上展示「會動的產品」是非常強大的：
*   **Vibe Coding 的實力證明**：這顯示您不僅會設計 UI，還能利用 AI 工具（如我們現在做的）快速將想法轉化為可操作的 Prototype。這種「設計+開發」的跨領域能力在現代市場極具競爭力。
*   **透明度與信任**：雖然我們隱藏了 AI 的底層技能設定，但公開「代碼成果」能讓人看到您對細節的掌控，以及如何解決複雜問題（如 Chrome Extension 的開發）。

**總結：**
既然您不介意公開，那我們目前的做法（**Public + 嚴格授權 + 隱藏隱私資料夾**）就是最平衡的方案。這既能讓您展示「技能樹的擴張」，又能在法律上禁止別人隨意拿去賺錢。
